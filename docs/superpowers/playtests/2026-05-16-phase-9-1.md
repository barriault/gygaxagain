# Phase 9.1 turn-discipline playtest

Date: 2026-05-16
Environment: Local development (Rails 8.1 alpha, Anthropic API live)
Device: desktop browser
Campaign: The Ancient Tomb of Phandalin (seeded — see [`db/seeds.rb`](../../../db/seeds.rb))
Party: Aragorn (PC, Ranger), Caine, Fred, Patric (companions)
Scene: Scene 36 — Cemetery & Tomb Approach (carried over from prior session via Reset Scene)

## Outcome

**Pass.** The Phase 9.1 turn-discipline contract held end-to-end. The party completed seven turns of exploration including a player-declared action, a companion declare-batch, a narrator-issued dice request via inline chip, a successful roll, and a clean continuation narration that ended at the player's handoff. The composer re-enabled itself automatically after the continuation completed; the player never had to refresh the page during normal play.

This is the first playtest since 2026-05-15 where the loop did not break before the party had a chance to act, and the first end-to-end demonstration that the dice-chip → roll → continuation cycle works.

## Session log

Started at scene 36 mid-flow (the scene had carry-over events from earlier same-day debugging sessions; phase started at `:awaiting_roll` after a manual Reset Scene during fixture cleanup). Player issued the perception declaration; narrator responded with companion-voiced beats (Caine, Fred, Patric each with a single appropriate reaction), then stopped at a dice chip: `[[1d20+5 — Aragorn Perception check]]`.

Player clicked the chip. Roll resolved (1d20 = 3, +5 = 8). NarrationJob fired a continuation. The continuation narrative landed: a sensory-uncertainty beat tied to the low roll, companion non-results, and a clean handoff (`What does Aragorn do?`). Composer re-enabled itself once the continuation completed. Verdict reached after a clean turn boundary, without refresh.

## Findings — what worked

- **PC/companion asymmetry held.** Aragorn (the PC) was never voiced or decided-for by the narrator. Companion lines and decisions read in-character without crossing into PC territory. No `[player_action @ ts]` or other prompt-format tags leaked into output.
- **Single-beat turn discipline held.** Each narration was one beat, ending in either a chip (awaiting roll) or a `?` handoff (awaiting declaration). No multi-PC-turn megablobs of the kind that broke the 2026-05-15 playtest.
- **Inline dice chips worked.** After today's fixes (`60bd69c`, `4509f11`, `30fb064`, `e92cece`, `407d689`, `ccc42fc`), the chip renders, clicks dispatch a POST, the roll lands as a `dice_roll` event with `turn_number` stamped, and the continuation narration fires automatically.
- **Markdown rendered correctly.** Paragraph breaks, em-dashes, and italics all rendered as intended via the Commonmarker pipeline in `NarrationComponent`.
- **Server-side chrome stayed in sync.** The composer's disabled state and the state-indicator's "waiting on …" message updated themselves over Turbo broadcasts as phase changed. No manual refresh required after the first fix landed.

## Findings — bugs found and fixed during the playtest

Six bugs surfaced and shipped during this session. All have regression coverage.

| Commit | Bug |
|---|---|
| `60bd69c` | NarrationJob persisted truncated chip text. Anthropic strips matched stop sequences from the streamed text; the saved narration ended `…Perception check` with no closing `]]`. ChipParser silently failed to recognize the chip, so the player saw raw text instead of a button. Fix: re-append the matched stop sequence in `finalize_success`. |
| `4509f11` | Dice-chip click handler aborted silently. `dice_chip_controller.js` uses `closest("[data-scene-id]")` to find the POST endpoint; the IDs lived on the composer (a sibling of the log subtree), so the lookup returned null. Fix: move `data-scene-id` and `data-campaign-id` to the play wrapper, where they ancestor both the log and composer. |
| `30fb064` | Continuation narration always 400'd on first attempt. `DiceRollsController` creates an empty streaming narration placeholder before enqueuing the continuation; `partial_turn_messages` then picked it up as the last narration and emitted an empty assistant message. The `-2` cache breakpoint landed on the empty block, triggering `cache_control cannot be set for empty text blocks`. Fix: filter empty narrations in `partial_turn_messages`. |
| `e92cece` | A single failed continuation permanently bricked the dice chip. `last_narration` in `SceneStateViewModel` returned the errored event (text=""); empty text doesn't match `OPEN_CHIP_RE`, so phase fell out of `:awaiting_roll` and `DiceRollsController` stopped enqueueing continuations. Fix: skip errored narrations in `last_narration`. |
| `407d689` | Composer stayed disabled after a continuation narration completed. The composer and state-indicator render with the phase-at-load disabled state and were never re-broadcast on phase change, so the player surface stayed in `:awaiting_roll` chrome until a manual refresh. Fix: new `Player::ChromeBroadcaster` fires on `Event.after_create_commit` and on `NarrationJob` finalize; state-indicator restructured to always render a stable wrapper div. |
| `ccc42fc` | Dice rolls were created without `turn_number`, silently dropping them from `completed_turn_messages` in subsequent prompts. Narrator would forget rolls between turns. Fix: stamp `turn_number` on the dice_roll in `DiceRollsController`. |

## Findings — surfaced after the initial PASS verdict

After the seven-turn dice-chip session completed and the verdict was reached, play continued for one more turn and surfaced a major content-fidelity failure that is structural, not a Phase 9.1 regression. Captured here so it isn't lost.

### Scene-boundary hallucination (manifestation of #16)

**What happened.** The party finished Captain Aldridge's briefing and declared movement into the tomb. The narrator continued into an "antechamber" scene that does not exist in scene 36's source material. Over the next four turns, the narrator invented a forced door, a scraping sound, and finally — on turn 8 — a room containing **one** wounded skeleton and **one** person hunched in the corner. That composite is a hallucination: scene 36 ("Cemetery & Tomb Approach") describes only the cemetery and the captain handoff. Scene 37 ("The Tomb — Entrance Hall") contains 2 skeletons at the left door trying to break through. Scene 38 ("The Tomb — West Side Chamber") contains Rewalt Mason locked alone inside. The narrator merged fragments of #37 and #38 into a single half-imagined room, with #37's skeleton count reduced from 2 to 1 and the door layout wrong.

**Root cause.** Two compounding factors:

1. **#16 — no scene transition mechanism.** The narrator's prompt only ever contains the *current* scene's secrets. Once the party crosses a scene boundary, the narrator has no source material for the next location and ad-libs from training-data inference. This issue is already scheduled for Phase 9.3.
2. **System-prompt discipline gap.** The asymmetry contract in [`system_prompt.rb`](../../../app/lib/narrator/system_prompt.rb) covers *querying* within a scene ("default to 'you find nothing remarkable' — do not invent") but says nothing about the party *moving past the scene's spatial scope*. The narrator interpreted "stealthily enters the tomb" as continuous narration rather than as a boundary-crossing event that should halt.

**Within-scene fidelity was perfect.** The captain conversation, briefing, tomb-door unbarring, and cover-story behavior all stayed on-source. Asymmetry held — no NPC secrets leaked, no PC dialogue was invented. The failure is exclusively at the scene boundary, not within scenes.

**Impact on Phase 9.1 verdict.** This does not invalidate the turn-discipline PASS — turn discipline, single-beat narration, dice chips, and asymmetry all held even during the hallucinated turns. But it confirms that "first end-to-end *playable* session" remains gated on Phase 9.3 landing. The 2026-05-15 log noted the same root cause as finding F10/#16; today it surfaces as the narrator inventing scene content rather than mis-attributing event timeline.

**Next-step suggestion** (not committed to without discussion): once Phase 9.3 lands the scene-transition affordance, also add scene-boundary discipline language to the system prompt — e.g. "If the party declares movement into a location not described in the current scene's notes, stop and ask the DM to set the next scene. Do not invent the new location's contents."

## Findings — known issues, deferred

- **Spec suite has 33 failing examples (of 662).** The failures are pre-existing stale specs from architectural changes already shipped on this branch: the Haiku-based `Narrator::DeclarationLlm` replacing the regex parser (`DeclarationParser` specs), the SceneAudit subsystem (`SceneAuditJob`, `AuditPromptBuilder`), `PcDeclarationsController` after the Haiku rewrite, and system specs from phases 6/7/8 referencing the old play surface. None of these are regressions from today's changes. Suite triage is a separate follow-up.
- **Declaration parser is overconfident on ambiguous input.** Mid-session, the parser converted a meta-game player message ("why are we doing X?") into a clarification prompt three times before the player typed an actual action. The Haiku-based parser is correctly rejecting non-actions, but the message back to the player could be more helpful. Not a blocker; observed earlier today in this scene's log.
- **Pre-existing two-test assertion bugs.** `gm_collection_prompt_component_spec.rb:8` and `narration_component_spec.rb:64` (dice chip class assertion) both expect literal strings the components no longer produce verbatim. Mechanical fixes, not Phase 9.1 work.
- **Rubocop**: 3 minor offenses in `declaration_parser_spec.rb` (all `Layout/SpaceBeforeFirstArg`, all autocorrectable). Will be cleaned up with the broader stale-spec triage since that file is part of the retired parser.

## Reflection

The 2026-05-15 playtest produced 13 findings (F1–F14). Phase 9.1's design spec and the 30-task implementation plan addressed the load-bearing blockers (F1, F3, F4, F5). Today's session validated all four of those fixes in a single live run, plus surfaced six new dice-chip-related bugs that did not exist in the prior session (the chip feature itself was new this phase).

The Phase 9.1 acceptance gate stated: "*the playtest produces a coherent end-to-end session, the narrator stops at the player's turn, dice integrate naturally, and markdown renders.*" The latter three held cleanly; "coherent end-to-end session" held *within* the bounds of scene 36 and broke the instant the party crossed a scene boundary — see the scene-boundary hallucination finding above. That failure is a manifestation of #16 (scheduled for Phase 9.3), not a Phase 9.1 regression.

Phase 9 ([issue #10](https://github.com/barriault/gygaxagain/issues/10)) — the blocker bucket from 2026-05-15 — is ready to close on the six items verified in this playtest. The remaining structural items (#16 scene transitions in Phase 9.3, #17 admin CRUD, #18 Solid Queue) gate "first end-to-end playable session" but not the Phase 9.1 turn-discipline contract.
