# v2 Phase 0 — Rails pivot roadmap

Date: 2026-05-13
Status: Roadmap design. Each subsequent phase (1–9) gets its own design spec when brainstormed.

## Purpose

v1 of gygaxagain proved the asymmetry mechanic — Claude-narrated solo D&D where hidden world state is denied to the narrator by construction, mediated only through scoped subagents and an MCP server. The proof-of-concept reached Phase 4b: dice routing, oracle routing, faction clocks with offscreen-developments tick, three-clue revelations, Mythic threads, module/lore intake, and a six-check bookkeeper audit at session-end.

Three forces motivate the pivot to v2 as a Rails web application:

1. **Personal play, durably.** v1 is locally-bound to one machine with a fragile Claude Code permission cascade. An upstream Claude Code regression ([anthropics/claude-code#25200](https://github.com/anthropics/claude-code/issues/25200)) currently prevents custom subagents from reaching MCP servers, materially degrading every dm/-touching feature. Even when the regression resolves, the engine remains tied to a specific Claude Code version on a specific machine with a venv that has to exist for the MCP to start. That's not a sustainable platform for actual play.

2. **A stronger asymmetry boundary.** v1's asymmetry is enforced by permission denies — the narrator "has tools it isn't supposed to use." v2 moves to asymmetry-by-context-construction: hidden state is never in the prompt sent to Claude, because the player-facing context builder doesn't have access to it. The narrator literally cannot leak what it was never given. This is a testable property of the code (assert: rendered prompt does not contain hidden field X) rather than a discipline the LLM is asked to honor. Strictly stronger guarantee.

3. **Productizable shape.** The repo owner has been a DM for years, is burned out, doesn't want to be the DM, and has no group member willing to take over. The engine *is* the path back to playing. A web app deployed to Heroku — accessible from desktop, tablet, phone, with proper authentication and durable persistence — is the only shape that makes that real. The fact that it could later become a real product (if it ever leaves alpha) justifies the rewrite scope.

v2 is not a port. The runtime architecture is fundamentally different (Rails service objects + scoped queries vs. Claude Code subagents + MCP). The gameplay design carries forward (the design specs in `docs/superpowers/specs/` describe behavior in runtime-agnostic terms); the code does not.

## Definition of done — v2 playing-MVP

v2 is "done enough to play on" when the following are true:

- A logged-in user can author a campaign (with at least Factions and NPCs) via the admin surface at `admin.gygaxagain.com`.
- The user can start a play session at `gygaxagain.com` and engage in turn-by-turn play: declared action → narration response → optional dice roll → optional oracle query → events log accumulates.
- Narration is generated by the Anthropic API via streaming, with the player-facing context constructed in a way that excludes hidden state.
- Asymmetry tests pass: no player-facing prompt builder, ViewModel, or rendered component leaks hidden columns from `*_secrets` tables.
- API costs are logged per call in the `llm_calls` table; a basic usage dashboard exists in admin.

This is **Phase 9 of v2**. Future phases beyond playing-MVP reintroduce v1 features (faction offscreen tick, revelations + three-clue rule, threads, module/lore intake) as additive phases. Each is its own brainstorm → spec → plan → implement cycle when prioritized.

What playing-MVP explicitly does NOT include (deferred to post-MVP phases):

- Faction clocks ticking offscreen between sessions (v1 Phase 2a equivalent).
- Revelation tracking with three-clue rule (v1 Phase 2b equivalent).
- Mythic thread CRUD (v1 Phase 2c equivalent).
- Mythic random-event composition (v1 Phase 2d equivalent).
- Module intake / lore intake (v1 Phase 3 equivalent).
- Session-end bookkeeper audit (v1 Phase 4a/4b equivalent).
- Multi-user sharing of campaigns. v2 is single-user-per-campaign forever; no invite, no role split, no "watch" mode.
- Stripe / subscriptions / free trials. Manual user creation only.
- Mobile-specific app. Responsive web (Tailwind) only.
- Provider-parity LLM. Anthropic is the default; the abstraction supports per-job provider override so Gemini can handle long-context intake later.

## Repo evolution from v1 to v2

The repo is preserved (not forked or re-initialized). v1 → v2 happens in-place:

1. **Tag v1 final state.** `git tag v1-final-poc` on the current `main`, pushed to origin. Provides cheap, durable retrievability of every v1 artifact via `git checkout v1-final-poc`.
2. **Delete v1 runtime.** `.claude/`, `CLAUDE.md`, `dm/`, `tools/dm-fs-mcp/`, `.mcp.json`. These are Claude Code-specific and have no role in v2.
3. **Keep design history and content.** `docs/superpowers/specs/` (the v1 specs become design lineage for v2); `world/`, `party/`, `sessions/`, `library/lore/`, `references/` (content/data that v2 will eventually import into Postgres via a future intake phase).
4. **Initialize Rails at repo root.** Standard `app/`, `config/`, `db/`, `Gemfile`, etc. The existing `docs/` and content directories coexist alongside.
5. **README rewrite as part of Phase 1 deliverables.**

The v1 final-poc tag is the safety net. Once tagged, the working tree can be cleaned without anxiety.

## Architectural commitments

These decisions are locked at Phase 0 and propagate through all v2 phases. Changes require a Phase 0 amendment (a follow-up spec, not silent drift).

### Stack

- **Rails 8.x latest stable.** Native authentication generators available but bypassed in favor of Devise (see Auth section).
- **PostgreSQL** as the primary datastore. Heroku Postgres `essential-0` tier ($5/mo) in alpha; upgrade when production traffic justifies it.
- **Bun** for JavaScript bundling via `jsbundling-rails`.
- **Tailwind CSS** via `cssbundling-rails`.
- **Propshaft** as the asset pipeline.
- **ViewComponent** for all view composition (MVVC pattern with explicit ViewModels).
- **Lookbook** in development for ViewComponent previews.
- **Hotwire** (Turbo + Stimulus) for client-side interactivity. No inline JavaScript; all client logic lives in Stimulus controllers.
- **RSpec** for tests (request, system, model, component, service).
- **Devise** for authentication, with `:registerable` disabled.
- **Solid Queue, Solid Cache, Solid Cable** in the shared Postgres database. Acceptable on Heroku at alpha scale; revisit if/when DB contention becomes measurable.
- **`pay` gem and Stripe integration** are explicitly deferred. Manual user creation in alpha.

Development tooling: `factory_bot_rails`, `capybara`, `shoulda-matchers`, `bullet` (dev), `annotaterb` (dev), `brakeman`, `rubocop-rails-omakase`, `erb_lint`, `dotenv-rails` (dev).

CI: GitHub Actions running RSpec + Brakeman + RuboCop + erb_lint on push/PR.

### Domain split + subdomain routing

- **`gygaxagain.com`** — the play surface. Player-facing context only. No campaign authoring affordances.
- **`admin.gygaxagain.com`** — the campaign management surface. Authoring, intake, dossier review, cost dashboards, account settings.
- **DNS via Cloudflare.** Apex uses CNAME flattening to point at the Heroku DNS target. `admin` is a plain CNAME. Both terminate at the same Heroku app; subdomain constraint in Rails routing is what dispatches.
- **Cross-subdomain Devise session.** Cookie domain `:all` with `tld_length: 2` so the session covers `*.gygaxagain.com`.
- **Routes split into two files.** `config/routes/play.rb` (constrained `subdomain: ''`) and `config/routes/admin.rb` (constrained `subdomain: 'admin'`), drawn from `config/routes.rb`.
- **Controller namespaces** mirror routes: `Play::ApplicationController` and `Admin::ApplicationController`, both descending from `ApplicationController`. The admin base sets `before_action :authenticate_user!`.
- **ViewComponent namespaces** mirror controllers: `app/components/play/`, `app/components/admin/`.

The subdomain split is the load-bearing UX enforcement of the asymmetry boundary. The play surface and the admin surface are different URLs, different controller trees, and different component trees. They can't accidentally render each other's content; bleed would require crossing a namespace boundary, which is visible in a code review.

### Auth — Devise without signup

- **Devise** with modules: `:database_authenticatable, :recoverable, :rememberable, :validatable, :trackable, :timeoutable, :lockable`.
- **No `:registerable`** — sign-up routes are not registered. `devise_for :users, skip: [:registrations]`.
- **No `:confirmable`** — there is no signup, so no confirmation flow.
- **Manual user creation.** A `users:create` rake task (or `rails console`) creates accounts. Alpha is invite-only by absence of an invite mechanism.
- **Sign-in URL** lives at apex (`gygaxagain.com/users/sign_in`). The cross-subdomain cookie carries the session to admin.
- **After-sign-in redirect** logic (refined in Phase 3): user with a `last_played_campaign_id` → directly into that campaign's play surface; user with campaigns but no last-played → play surface campaign picker; user with no campaigns → admin new-campaign flow.
- **ActionMailer** configured to read SMTP from ENV vars but defaults blank in alpha. Password reset will silently fail until an email provider is wired. Acceptable for invite-only alpha.
- **Pundit** (or `action_policy`) is deferred. `before_action :authenticate_user!` plus `current_user.campaigns.find(params[:id])` scoping is sufficient until the rules get non-trivial.

### Multi-tenancy

- **Single-user-per-campaign.** A `User` `has_many :campaigns`; a `Campaign` `belongs_to :user`. No `Membership` model. No roles. No sharing between users, period.
- **Tenant scoping via the `current_user` association.** No `acts_as_tenant` gem in v2 — the gem buys discipline we can get from a `CampaignScoped` controller concern that loads campaigns through `current_user.campaigns.find(params[:id])` (404 on miss).
- **Default scope on cross-cutting models.** Anything with a `campaign_id` foreign key has a `belongs_to :campaign` and is reachable only through the campaign. The asymmetry tables (`factions`, `npcs`, `scenes`, etc.) all carry `campaign_id`.

### Asymmetry model — separate `_secrets` tables + ViewModels

The load-bearing invariant of the engine. Two-layer enforcement:

**Layer 1 — Data model.** Public-facing state and hidden state live in separate tables. Concretely:

- `factions` (public): name, public_description, faction-public attributes.
- `faction_secrets` (private): hidden clock, hidden motivation, secret connections, etc. `belongs_to :faction`.
- `npcs` (public): name, public_description, location.
- `npc_secrets` (private): true identity, hidden agenda, faction affiliation if hidden, etc.
- `revelations` (public): the player-facing hook text and known clue paths.
- `revelation_secrets` (private): the underlying reveal content, clue vector descriptions, trigger conditions.
- `modules` (public): metadata. `module_secrets` (private): the full module content that hasn't been revealed yet.
- `scenes`: public scene state.
- `events`: polymorphic scene events (narration, dice_roll, oracle_query, revelation_delivered, scene_transition, etc.). Always player-visible at the time they happen; never contain hidden state that wasn't already revealed in-fiction.

**Layer 2 — ViewModel discipline.** Two parallel ViewModel namespaces:

- `Player::FactionViewModel` — wraps a `Faction` for player-facing rendering. Never reaches across the `belongs_to :faction_secret` join. Cannot return hidden attributes by construction.
- `Narrator::FactionViewModel` — wraps a `Faction` + its `FactionSecret` for narrator/admin-side rendering. Can return everything.

Same pattern for `NpcViewModel`, `RevelationViewModel`, etc.

**Layer 3 — Prompt builder discipline.** The class that constructs Claude API prompts for player-facing narration (`Narrator::PromptBuilder` — name preserved for clarity) takes only Player-namespace ViewModels as input. The narrator-side ViewModels are not reachable from the player-narration code path. Tests assert this.

**Asymmetry test surface.** A custom RSpec matcher: `expect(prompt).not_to_leak(hidden_attrs_of: faction)`. Every player-facing ViewModel spec asserts that hidden fields are not exposed. Every prompt builder spec asserts the rendered prompt does not contain values from `*_secrets` tables. This is the discipline that replaces v1's "the narrator is instructed not to read dm/" with "the narrator literally has no access path to hidden state."

### LLM provider strategy

- **`Llm::Provider`** is the abstraction. Concrete adapters: `Llm::Providers::Anthropic` (primary), with provider selection via job type — `Llm::Provider.for(:narration)` returns Anthropic; `Llm::Provider.for(:intake_long_context)` returns Google's Gemini (when wired in a future phase).
- **Streaming is first-class.** Narration calls stream tokens. Turbo Streams + ActionCable + a Stimulus controller render the streamed response live in the play UI scene log.
- **Prompt caching is exposed per-provider.** Anthropic's explicit cache_control breakpoints are surfaced through the adapter; other providers' implicit caching is documented but not configurable.
- **No `langchainrb`, no `ruby_llm`, no heavyweight abstraction.** Thin per-provider adapters owned in our codebase.
- **API keys via ENV.** `ANTHROPIC_API_KEY`, `GOOGLE_API_KEY`, etc. Not committed; not in version control; rotated via Heroku config.

### Cost tracking — `llm_calls` table from day one

Every API call writes a row to `llm_calls`. Schema:

```ruby
create_table :llm_calls do |t|
  t.references :user, null: false
  t.references :campaign, null: true       # null for cross-campaign admin work
  t.references :scene, null: true          # null for out-of-session calls
  t.string :purpose, null: false           # 'narration', 'oracle_compose', 'bookkeeper_audit', 'intake_extract', etc.
  t.string :provider, null: false          # 'anthropic', 'google', 'openai'
  t.string :model, null: false             # 'claude-sonnet-4.5', 'gemini-2.5-pro', etc.
  t.integer :input_tokens, default: 0
  t.integer :output_tokens, default: 0
  t.integer :cache_creation_tokens, default: 0   # Anthropic-specific
  t.integer :cache_read_tokens, default: 0       # Anthropic-specific
  t.integer :total_cost_cents, default: 0
  t.integer :latency_ms
  t.string :provider_request_id            # for support tickets
  t.jsonb :prompt_payload                  # full prompt sent to provider
  t.jsonb :response_payload                # full response received
  t.timestamps
end
```

- **`Llm::Pricing.for(provider:, model:)`** returns rates keyed on token category. Computed at write-time so dashboard queries are fast.
- **Admin dashboard** at `admin.gygaxagain.com/usage` shows daily/weekly rollups by campaign, by purpose, by model. Per-campaign "lifetime cost" widget on the campaign index in admin.

### Testing discipline

- **RSpec** for all tests. Request specs, system specs (Capybara + Selenium), model specs, component specs (ViewComponent's test helpers), service specs.
- **Asymmetry tests are a project-wide convention.** Any class that constructs a player-facing prompt, ViewModel, or component template MUST have a spec asserting it does not leak hidden fields. Enforced by code review; ideally also by a Brakeman-like custom analyzer in a later phase.
- **Coverage targets** are not enforced numerically. Coverage tools (SimpleCov) optional.
- **Factories over fixtures** for test data; `factory_bot_rails` is included from Phase 1.

## Phase progression

Each phase below has: a one-paragraph scope statement, its dependencies, its acceptance criteria. Phases 1 and 2 also have detailed scopes (below). Phases 3–9 get their own design specs when brainstormed individually.

Each phase corresponds to one GitHub sub-issue under the v2 parent issue. The phase is "complete" when its acceptance criteria pass and its sub-issue is closed. Closed sub-issues link to their per-phase design spec(s) and the implementation PR(s).

### Phase 1 — Rails skeleton + landing page on Heroku

Bootstrap the v2 codebase. After this phase, `https://gygaxagain.com` resolves to a Rails 8 app on Heroku showing a placeholder landing page, with CI running RSpec on every push. No auth, no data models, no LLM integration.

Dependencies: none.

Acceptance criteria:
- Local: `bin/rails server` works; `bundle exec rspec` passes ≥1 test.
- CI: GitHub Actions workflow green on `main`.
- Production: `https://gygaxagain.com` resolves to the Heroku app, renders the landing page, valid SSL.
- v1 cleanup complete: `v1-final-poc` tag exists at origin; `.claude/`, `CLAUDE.md`, `dm/`, `tools/dm-fs-mcp/`, `.mcp.json` removed from working tree.
- README rewritten to reflect v2.

### Phase 2 — Devise + subdomain split + admin shell

Add authentication and the URL-space split. Sign-in works at apex; admin subdomain requires auth and exists as an empty shell. No data models yet.

Dependencies: Phase 1.

Acceptance criteria:
- Sign-in at `https://gygaxagain.com/users/sign_in` works; sign-up returns 404.
- Cross-subdomain session: signed-in user can navigate between apex and `admin.gygaxagain.com` without re-auth.
- `https://admin.gygaxagain.com` resolves, requires auth, shows an empty dashboard placeholder.
- Manual user creation works via `rake users:create` task.
- All tests green; CI green.

### Phase 3 — User → Campaign + Campaign CRUD + auth redirect logic

Introduce the Campaign model and admin-side CRUD. User table grows a `last_played_campaign_id`. After-sign-in redirect lands the user appropriately (last-played → play; no campaigns → new-campaign flow).

Dependencies: Phase 2.

Acceptance criteria:
- Admin user can create, edit, delete campaigns from `admin.gygaxagain.com/campaigns`.
- After sign-in, redirect logic places user correctly per the three cases (last-played / has-campaigns / no-campaigns).
- Play surface at `https://gygaxagain.com/campaigns/:id/play` is accessible only when `campaign.user_id == current_user.id`; 404 otherwise.
- Campaign deletion cascades cleanly to all associated rows (will matter more in Phase 5+).
- Tests cover the auth redirect matrix and the campaign ownership check.

### Phase 4 — LLM provider abstraction + `llm_calls` + Anthropic adapter

Build the LLM client layer. Thin per-provider adapters under `Llm::Providers::`, a top-level `Llm::Provider.for(:purpose)` lookup, the `llm_calls` table, and a `Llm::Pricing` module. First end-to-end: a "test narration" admin tool that calls Anthropic and writes an `llm_calls` row. Streaming not required yet — that comes in Phase 8.

Dependencies: Phase 3.

Acceptance criteria:
- `Llm::Provider.for(:narration)` returns an Anthropic adapter.
- An admin tool at `admin.gygaxagain.com/diagnostics/llm` lets the user submit a prompt, see the response, see the `llm_calls` row written.
- `Llm::Pricing` returns correct rates for at least one Anthropic model (Sonnet) across input / output / cache_creation / cache_read.
- Tests stub the HTTP layer; no real API calls in CI.
- The `:intake_long_context` purpose is wired but returns Anthropic as a placeholder (Gemini adapter is a later phase).

### Phase 5 — Asymmetry schema + ViewModels + test infrastructure

The load-bearing data layer. Migrations for `Faction`/`FactionSecret`, `Npc`/`NpcSecret`, `Scene`, `Event` (polymorphic). Player and Narrator ViewModel namespaces, with a custom RSpec matcher (`not_to_leak`) for asymmetry assertions. No UI yet — these are bare models and ViewModels.

Dependencies: Phase 3 (Campaign exists).

Acceptance criteria:
- Migrations create all six tables with appropriate foreign keys, indexes, and `belongs_to :campaign` scoping.
- `Player::FactionViewModel` exposes only public fields; spec asserts it.
- `Narrator::FactionViewModel` exposes everything; spec asserts it.
- The `not_to_leak` custom matcher exists and is documented in `spec/support/matchers/`.
- Same pattern verified for `Npc`.
- Event model supports the polymorphic kinds: narration, dice_roll, oracle_query, scene_transition (more kinds added in later phases as needed).

### Phase 6 — Play surface UI shell

The chat-like UI for play. Scene log rendered as a list of polymorphic Event components. Scene CRUD in admin (so the user can create scenes to play through). Empty scenes render an empty log; no LLM yet, no dice yet — pure UI shell.

Dependencies: Phase 5 (Events exist).

Acceptance criteria:
- `https://gygaxagain.com/campaigns/:id/scenes/:scene_id/play` renders the scene log via `Play::SceneLogComponent`.
- Each Event kind has a corresponding `Play::Events::*Component` (NarrationComponent, DiceRollComponent, OracleQueryComponent, SceneTransitionComponent).
- A polymorphic dispatcher (`Play::Events::Component.for(event)`) routes each event to its component.
- Admin can create/edit/delete scenes from `admin.gygaxagain.com/campaigns/:id/scenes`.
- Lookbook previews exist for each Event component.

### Phase 7 — Dice + Mythic oracle service objects

Add the two non-LLM mechanical surfaces: dice rolls (with the existing v1 expression parser ported as a service) and Mythic GME 2e oracle resolution (with chaos factor). Both surface as buttons / forms in the play UI. Each invocation creates an Event in the scene log.

Dependencies: Phase 6.

Acceptance criteria:
- `Dice::Roll` service parses expressions like `2d6+3`, `1d20-1`, returns a structured result.
- `Mythic::Oracle` service takes a question + likelihood + chaos factor, returns yes/no/exceptional/random_event.
- Both are surfaced in the play UI; clicking creates an Event, broadcast via Turbo Streams to the scene log.
- Chaos factor is per-campaign state, adjustable from admin.
- Tests cover dice expression edge cases and oracle outcome tables.

### Phase 8 — Narrator integration + streaming

The big one. `Narrator::PromptBuilder` constructs player-facing prompts from `Player::*ViewModel` inputs (and only those). Streaming Anthropic call yields tokens; Turbo Streams broadcasts them to the scene log as a live-updating Narration event. Post-session audit job runs in Solid Queue (lightweight bookkeeper equivalent — narrator-discipline checks only, no asymmetry-violation detection since that's structural in v2).

Dependencies: Phases 4 (LLM client), 5 (ViewModels), 6 (play UI), 7 (dice + oracle).

Acceptance criteria:
- Player can submit an action; narration response streams in token-by-token.
- The prompt sent to Anthropic is reachable via the `llm_calls` row. Full prompt and full response are logged in `jsonb` columns (`prompt_payload`, `response_payload`) for debug + future analysis. PII concerns are minimal in a single-user solo app; storage retention is a later concern if total table size grows uncomfortable.
- The session-end audit job runs at session close, writes a structured audit result to the campaign.
- Asymmetry tests: every prompt built by `Narrator::PromptBuilder` is verified to not leak hidden state.

### Phase 9 — Asymmetry hardening + first end-to-end playable session

The shake-out phase. Comprehensive asymmetry test coverage across every player-facing surface. First real campaign played end-to-end on the v2 deployment. Bug-fixes and polish.

Dependencies: Phases 1–8.

Acceptance criteria:
- Every `Player::*` ViewModel has a spec asserting `not_to_leak` against its corresponding `*Secret` table.
- Every `Play::*Component` has a spec asserting it does not render hidden state when rendered with a Player ViewModel as input.
- The repo owner plays one full session (start → multiple scenes → multiple events → session-end audit) on the production deployment. Notes captured in a session log file (which v2 may or may not need to migrate from markdown — decided when this phase brainstorms).
- Issues uncovered during the shake-out are logged as new sub-issues; the v2 parent issue (Phase 0) does not close until shake-out passes.

## Future phases (post-MVP)

After playing-MVP, the v1 features come back as additive phases. Each is its own brainstorm → spec → plan → implement cycle. Likely shape:

- **Phase 10** — Faction clocks + offscreen-developments tick (v1 Phase 2a equivalent). Faction clocks live in `faction_secrets`; offscreen tick runs as a Solid Queue job between sessions.
- **Phase 11** — Revelations + three-clue rule (v1 Phase 2b equivalent). Revelations + clue vectors live in `revelations` + `revelation_secrets`; clue delivery tracked in events.
- **Phase 12** — Threads (v1 Phase 2c equivalent). Mythic threads with open/list/close CRUD.
- **Phase 13** — Mythic random-event composition (v1 Phase 2d equivalent). Random events thread through factions, revelations, threads.
- **Phase 14** — Module intake (v1 Phase 3a equivalent), Gemini-on-intake provider (long context for full module ingest).
- **Phase 15** — Lore intake + admin lookups (v1 Phase 3c equivalent).
- **Phase 16** — Revelation auto-proposals + faction auto-proposals from intake (v1 Phase 3d/3e equivalent).
- **Phase 17** — Session-end audit hardening (v1 Phase 4b equivalent — but structurally simpler in v2 because asymmetry violations are caught by tests, not by post-hoc audit).

The order is not locked. Phase 10 onwards prioritizes by what the user wants to play with next.

## Out of scope

Permanently or until-further-notice:

- **Multi-user campaigns.** No invite system, no roles, no "watch" mode.
- **Live multiplayer.** This is a solo engine for one player at a time.
- **Mobile native app.** Responsive web only.
- **Payments / subscriptions / free trials.** Alpha is invite-only; commercial phase is far future.
- **Provider parity.** Anthropic-default; Gemini for one specific job class (long-context intake); OpenAI not currently planned.
- **Real-time DM-style override controls** ("temporarily reveal X to the player"). The asymmetry boundary is not user-toggleable at runtime.
- **AI-assisted character creation / leveling.** Players bring their own character sheet (manually entered).
- **VTT integration** (Roll20, Foundry, etc.). Out of scope.
- **Voice / audio narration.** Text only.

## Workflow — issues + per-phase loop

This Phase 0 spec defines the roadmap and the architectural commitments. Each subsequent phase has its own design lifecycle:

1. **GitHub issue** created from this Phase 0 spec — parent issue (`v2 Rails pivot — Phase 0`) + 9 sub-issues (one per phase). Each sub-issue body has a scope statement, dependencies, acceptance criteria, and a TBD link for the per-phase design spec.
2. **When a phase is starting:** the sub-issue is referenced (e.g., "Brainstorm Phase 3") which kicks off a brainstorming session per the superpowers workflow. The brainstorm produces a per-phase design spec written to `docs/superpowers/specs/YYYY-MM-DD-v2-phase-N-<topic>-design.md`.
3. **The per-phase spec drives a writing-plans pass,** producing an implementation plan at `docs/superpowers/plans/YYYY-MM-DD-v2-phase-N-<topic>-plan.md`.
4. **Implementation** follows the plan. PRs reference the sub-issue; tests/CI green required to merge.
5. **Sub-issue closes** when acceptance criteria pass; spec and plan paths are added to the sub-issue body before closing.

The v2 parent issue does not close until Phase 9 ships and the first end-to-end session has been played.
